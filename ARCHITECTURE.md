# Architecture

## System Overview

**KiloWhat?** is a React Native mobile application with a simple three-layer architecture:
1. **UI Layer** - React Native screen components
2. **Storage Layer** - AsyncStorage data access functions
3. **Data Layer** - TypeScript type definitions

There is no backend, no state management library, and no complex data flow. All state is either local to components (useState) or persisted to device storage (AsyncStorage).

## Data Flow

### Reading Data (App Startup / Screen Focus)
```
Screen Component
  ↓ (useEffect on mount/focus)
Storage Function (loadCars, loadSettings)
  ↓ (async/await)
AsyncStorage.getItem()
  ↓ (JSON.parse)
Update Component State (setState)
  ↓
Re-render UI
```

### Writing Data (User Actions)
```
User Interaction (button press, form submit)
  ↓
Validation (Alert.alert if invalid)
  ↓ (if valid)
Storage Function (addCar, updateCar, saveSettings)
  ↓ (async/await)
AsyncStorage.setItem()
  ↓
Navigate Back or Update State
  ↓
Previous Screen Re-fetches Data (focus listener)
```

### Calculation Flow (Real-time)
```
User Input (TextInput onChange)
  ↓
Update State (setCurrentSoc, setTargetSoc)
  ↓
Re-render Triggers
  ↓
Derived Values Recalculated
  ↓
Display Updated Results
```

## Key Modules

### Screens (src/screens/)

#### CalculatorScreen.tsx
**Purpose:** Main application screen for calculating charging time

**Key Features:**
- Car selector (pill-style buttons)
- Current/target SOC input fields (numeric)
- Real-time kWh and time calculation
- Navigation to car management and settings
- Auto-focus on current charge field (optional)
- Persists last entered current charge value

**State:**
- `cars` - List of all user cars
- `selectedIndex` - Index of currently selected car
- `currentSoc` - Current state of charge input
- `targetSoc` - Target state of charge input
- `efficiency` - Charger efficiency from settings
- `maxOutputKw` - Charger max output from settings

**Calculations:**
```typescript
kwhNeeded = (batteryCapacityKwh * (target - current) / 100) / (efficiency / 100)
chargeTime = kwhNeeded / maxOutputKw
```

#### CarManagementScreen.tsx
**Purpose:** List all saved cars with edit/delete actions

**Key Features:**
- Display all cars with name and battery capacity
- Swipe-to-delete gesture (or delete button)
- Navigate to AddEditCar for editing
- Navigate to AddEditCar for adding new car
- Empty state prompts user to add first car

#### AddEditCarScreen.tsx
**Purpose:** Form to create new car or edit existing car

**Key Features:**
- Text input for car name
- Numeric input for battery capacity (kWh)
- Numeric input for default target SOC (%)
- Validation before saving (required fields, valid ranges)
- Generate UUID for new cars
- Navigate back on successful save

**Validation Rules:**
- Name: Required, non-empty after trim
- Battery capacity: Number > 0
- Default target SOC: Number 0-100

#### SettingsScreen.tsx
**Purpose:** Configure global app settings

**Key Features:**
- Charger efficiency input (default 90%)
- Charger max output input (default 7.4 kW)
- Auto-focus current charge toggle
- Save settings on change
- Load settings on mount

### Storage Layer (src/storage/)

#### cars.ts
**Purpose:** CRUD operations for Car entities

**Functions:**
- `loadCars()` - Returns array of all cars from AsyncStorage
- `saveCars(cars)` - Overwrites entire car array
- `addCar(car)` - Appends new car to array
- `updateCar(car)` - Replaces car with matching ID
- `deleteCar(id)` - Removes car with matching ID

**Storage Key:** `'evapp_cars'`

**Error Handling:** None (fail fast, crash on error)

#### settings.ts
**Purpose:** Load and save app settings

**Functions:**
- `loadSettings()` - Returns settings, merged with defaults
- `saveSettings(settings)` - Overwrites entire settings object

**Storage Key:** `'evapp_settings'`

**Default Values:**
```typescript
{
  chargerEfficiency: 90,
  chargerMaxOutputKw: 7.4,
  autoFocusCurrentCharge: false,
  lastCurrentSoc: 20
}
```

### Type Definitions (src/types/)

#### Car.ts
```typescript
interface Car {
  id: string;                    // UUID v4
  name: string;                  // User-defined name
  batteryCapacityKwh: number;    // Usable battery capacity
  defaultTargetSoc: number;      // Default target charge (0-100)
}
```

#### Settings.ts
```typescript
interface Settings {
  chargerEfficiency: number;        // Percentage (0-100)
  chargerMaxOutputKw: number;       // Kilowatts
  autoFocusCurrentCharge: boolean;  // Auto-focus current SOC field
  lastCurrentSoc: number;           // Last entered current SOC (0-100)
}
```

#### navigation.ts
```typescript
type RootStackParamList = {
  Calculator: undefined;
  CarManagement: undefined;
  AddEditCar: { car?: Car };  // Optional car for editing
  Settings: undefined;
}
```

## Navigation Structure

```
Calculator (Home)
├── → CarManagement
│   └── → AddEditCar (modal)
│       ├── Add new car (no params)
│       └── Edit car (car param)
```

**Navigation Pattern:**
- Stack navigator with native transitions
- Type-safe params via `RootStackParamList`
- Screens refetch data on focus (using `navigation.addListener('focus')`)
- Modal behavior for AddEditCar (slides up on iOS, standard on Android)

## State Management

### No Global State
This app intentionally avoids Redux, MobX, Zustand, or any global state library.

**Rationale:**
- Simple app with limited shared state
- AsyncStorage serves as source of truth
- Screens refetch data when focused
- No complex state synchronization needed

### Component-Level State
- **useState** - UI state (inputs, selected indices)
- **useRef** - Non-visual state (flags, focus refs)
- **useEffect** - Side effects (data loading, focus listeners)
- **useCallback** - Memoized functions passed to children or deps

### Data Persistence
- **AsyncStorage** - Device-local key-value store
- One storage module per domain (cars, settings)
- Simple CRUD functions with Promise-based API
- No caching layer (always read from storage)

## Styling Approach

### Dark Theme
All screens use a consistent dark theme:
- Background: `#111` (dark gray, almost black)
- Cards: `#1a1a1a` (lighter gray for elevation)
- Borders: `#333` (subtle borders)
- Text: `#fff` (white), `#aaa` (gray), `#555` (muted)
- Primary: `#00c896` (teal/green accent)

### Layout
- **Spacing:** Multiples of 4 (4, 8, 12, 16, 20, 24, 32)
- **Border radius:** 12-20px for cards, 8-12px for inputs
- **Typography:** System fonts, weights 200-600
- **Icons:** Material Design Icons via vector-icons

### Responsive Design
- No fixed widths (use flex and percentages)
- Safe area handling via `SafeAreaProvider`
- Keyboard avoidance on iOS with `KeyboardAvoidingView`
- ScrollViews with `keyboardShouldPersistTaps="handled"`

## Error Handling Strategy

### Validation Errors
- User input validated before async operations
- `Alert.alert()` for validation failures
- Guard clauses for early returns
- No form submission if validation fails

### Storage Errors
- No try/catch in storage layer
- Let errors propagate to crash reporting (fail fast)
- Assumption: AsyncStorage operations should always succeed
- If storage fails, app is in unrecoverable state

### Type Safety
- TypeScript strict mode catches most errors at compile time
- Explicit types for all function parameters and returns
- No `any` types allowed
- Optional chaining for potentially undefined values

## Performance Considerations

### Optimization Strategies
- `useCallback` for functions in dependency arrays
- `useRef` to avoid re-renders for non-visual state
- Derived values computed inline (no useMemo needed yet)
- Navigation focus listeners prevent stale data

### Not Optimized Yet
- No virtualization (car lists expected to be small)
- No memoization (calculations are trivial)
- No code splitting (app is small)

### Performance Constraints
- Small data sets (dozens of cars, not thousands)
- Simple calculations (no heavy computation)
- No image processing or large assets
- Target: 60fps on mid-range devices

## Build and Deployment

### Local Development Builds
- `expo run:android` - Build and run on Android device/emulator
- `expo run:ios` - Build and run on iOS simulator/device
- `expo start --dev-client` - Start dev server for dev builds

### No EAS or Expo Go
- Local native builds only
- Supports custom native modules
- Full control over native code
- No cloud build service dependency

### Platform Targets
- **iOS:** Minimum iOS 13.4 (React Native 0.83 requirement)
- **Android:** Minimum SDK 23 (Android 6.0)
- **Web:** Modern browsers (Expo web support)

## Security Considerations

### No Authentication
- No user accounts
- No API keys or secrets
- No sensitive data stored

### Data Privacy
- All data stays on device
- No network requests
- No analytics or tracking
- No third-party SDKs (except UI libraries)

### Input Validation
- Numeric inputs sanitized (parseFloat, parseInt)
- Range validation (0-100 for percentages)
- String trimming for text inputs
- No SQL injection risk (no database)
- No XSS risk (no web content rendering)
